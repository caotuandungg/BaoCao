Biên bản bàn giao hệ thống giám sát KubernetesMôi trường Production (DC CMC)1. Mô hình hệ thống giám sát/Topology này mô tả kiến trúc của Kube-Prometheus-Stack trong môi trường Kubernetes, thể hiện đầy đủ pipeline từ thu thập metrics → lưu trữ → truy vấn → cảnh báo.Monitoring Targets – nơi phát sinh dữ liệu:Node Exporter thu thập metrics từ host/OS (CPU, RAM, disk…) Kube-state-metrics cung cấp trạng thái tài nguyên Kubernetes (pod, deployment…) Prometheus Exporter và Instrumented Application expose metrics từ ứng dụng Các nguồn này không push dữ liệu, mà được Prometheus pull metrics theo mô hình scrape.Prometheus Server (chạy trong Pod):Service Discovery kết nối với Kubernetes API để tự động phát hiện target (pods, services) Scraping Config định nghĩa cách scrape (interval, endpoint…) Retrieval thực hiện pull metrics TSDB (Time Series Database) lưu trữ dữ liệu dạng time-series HTTP Server cung cấp API/query cho bên ngoài Toàn bộ cấu hình scrape không hardcode, mà được quản lý động thông qua CRDs.Prometheus Operator - Là control plane quản lý Prometheus stack thông qua Kubernetes CRDs như: Prometheus ServiceMonitor / PodMonitor AlertManager PrometheusRule ScrapeConfig Operator watch các CRD và tự động generate/config lại Prometheus và Alertmanager Visualization & Alerting:Grafana:Query dữ liệu từ Prometheus Hiển thị dashboard (metrics visualization) Chạy dưới dạng Pod trong cluster Alerting flow:Prometheus evaluate rules (PrometheusRule) Khi match condition → push alerts sang Alertmanager Alertmanager: Deduplicate Group alerts Route theo config Sau đó Alert được gửi ra ngoài qua Webhook (Telegram trong sơ đồ)Hệ thống giám sát này được triển khai theo mô hình tập trung, sử dụng bộ công cụ Kube-prometheus-stack (Helm Chart rancher-monitoring version 106.0.1_up66.7.1-rancher.10) quản lý qua Rancher tại k8s.vnpost.vn (Cluster ID: c-gjbfb).Thông tin Cluster:Platform: RKE2 v1.31.7+rke2r1 trên Ubuntu 22.04.5 LTSTopology: 3 Master nodes (Control Plane + etcd) + 6 Worker nodes = 9 nodesContainer Runtime: containerd 2.0.4-k3s2 / 2.0.5-k3s1Namespace giám sát: cattle-monitoring-system (37 pods), cattle-dashboards (34 dashboard ConfigMaps)Các hệ thống thành phần (Thu thập và xử lý):Prometheus v2.55.1: Core engine chịu trách nhiệm scrape (thu thập) metrics từ các node, pod, và dịch vụ trong cụm K8S. Lưu trữ dữ liệu dạng time-series. Scrape interval: 30s, Evaluation interval: 30s, Retention: 10d / 50GiB.Prometheus Operator v0.79.0: Quản lý lifecycle của Prometheus và Alertmanager, tự động reload cấu hình khi có thay đổi CRDs.Node Exporter v1.3.1: Cài đặt trên mỗi Node K8S (DaemonSet — 9 pods) để thu thập tài nguyên phần cứng (CPU, RAM, Disk, Network).Kube-state-metrics v2.14.0: Giao tiếp với Kube-API để thu thập các thông tin về trạng thái của các object K8S (Deployments, Pods, StatefulSets, DaemonSets...).Alertmanager v0.27.0: Tiếp nhận cảnh báo từ Prometheus, thực hiện gom nhóm (grouping), chống nhiễu (inhibition), và định tuyến (routing) gửi thông báo ra các kênh. Retention: 120h.PushProx (Client/Proxy): Thu thập metrics từ các K8S core components (etcd, scheduler, controller-manager, kube-proxy) qua mô hình reverse proxy — tổng cộng 4 cặp DaemonSet/Deployment.Prometheus Adapter v0.12.0: Cung cấp Custom Metrics API cho HPA (Horizontal Pod Autoscaler).Hệ thống tập trung/Báo cáo:Grafana v11.1.0: Hệ thống dashboard tập trung kết nối với Prometheus. Cung cấp 34 dashboards báo cáo trực quan về tình trạng hạ tầng, cluster và ứng dụng.2. Phạm vi chuyển giao giám sát & Monitor Item List BaselinePhân định rõ rệt phạm vi giám sát giữa đơn vị quản lý hạ tầng (Cloud) và người dùng sử dụng dịch vụ K8S.Phạm vi của Cloud (Quản trị Hạ tầng & Cluster K8S):Giám sát các Node (Vật lý/Máy ảo): CPU, Memory, Disk IO, Disk Usage, Network Traffic.Giám sát Core K8S Components: Kube API Server, etcd, Kubelet, Scheduler, Controller Manager.Giám sát hạ tầng mạng/Ingress Controller.Phạm vi của Khách hàng (Đơn vị sử dụng K8S):Giám sát trạng thái Pod (Running, Pending, Failed, CrashLoopBackOff).Giám sát tài nguyên (CPU/RAM) tiêu thụ của từng Pod/Namespace.Giám sát Application Metrics (qua việc cấu hình ServiceMonitor/PodMonitor).Monitor Item List Baseline (Danh sách các item giám sát cơ bản):Hạ tầng: Node Not Ready, Node High CPU (> 80%), Node High Memory (> 80%), Disk Full (> 85%).Cluster: Kube API Down, etcd Member Down, CoreDNS Down.Ứng dụng: KubePodCrashLooping, KubePodNotReady, Deployment Generation Mismatch, OOMKilled.3. Vai trò, trách nhiệm (W/R/I) và Quy trình tích hợp dịch vụPhân chia vai trò dựa trên mô hình W/R/I (Write/Read/Inform) hoặc theo trách nhiệm xử lý.Đơn vị Cloud/Hạ tầng:Trách nhiệm: Vận hành hệ thống Kube-prometheus-stack lõi đảm bảo luôn hoạt động (HA). Xử lý các cảnh báo liên quan đến Node và K8S Core.Vai trò: Write (Cấu hình Prometheus/Alertmanager core), R/I (Xử lý lỗi hạ tầng).Khách hàng (Đơn vị phát triển/vận hành ứng dụng):Trách nhiệm: Tự vận hành ứng dụng, theo dõi dashboard Grafana, và xử lý khi có cảnh báo ứng dụng chết/quá tải.Vai trò: Read (Xem Grafana), Write (Tạo cấu hình ServiceMonitor, PrometheusRule cho namespace của mình).Đơn vị Điều hành dịch vụ (ĐHDV):Trách nhiệm: Nắm bắt tổng quan tình trạng dịch vụ, điều phối khi có sự cố diện rộng.Vai trò: Inform/Read.Cách thức thông tin cho ĐHDV khi add thêm dịch vụ mới vào K8S:Khách hàng/Đội Dev triển khai ứng dụng kèm theo CRD ServiceMonitor hoặc PodMonitor vào namespace tương ứng. Kube-prometheus-stack sẽ tự động phát hiện (auto-discover) và thu thập metrics.Đội Dev tạo ticket/thông báo qua email cho ĐHDV và Đội Cloud về tên dịch vụ mới, namespace, và endpoint metrics để ĐHDV cập nhật danh sách dịch vụ cần theo dõi trên Dashboard tập trung.Sau khi bàn giao, cập nhật thông tin chủ quản lên hệ thống External Alert4. Chuyển giao cảnh báoBàn giao bộ rules cảnh báo (PrometheusRules) đã được định nghĩa sẵn. Cảnh báo được chia thành các mức độ:Warning (Cảnh báo sớm): Cần lên kế hoạch kiểm tra.Ví dụ: CPU/RAM > 80% trong 5 phút, Disk usage > 80%, Pod restart > 3 lần/15 phút.Critical (Nghiêm trọng): Cần xử lý ngay lập tức gây gián đoạn dịch vụ.Ví dụ: Node Down, Kube API Down, CPU/RAM > 95% liên tục, Disk usage > 90%, Pod trạng thái CrashLoopBackOff liên tục.(Người dùng có thể tự override hoặc tạo thêm ngưỡng cảnh báo riêng cho ứng dụng của mình bằng cách tạo CRD PrometheusRule trong namespace của họ).Danh sách đầy đủ các Rule hiện đang có trong Cluster (33 Rule Groups)Để xem danh sách trên
